# Conventions vs nvdiffrast

This document captures the **two semantic conventions** where this library
deliberately differs from upstream nvdiffrast. Both differences are
*conventions*, not bugs — either library is internally consistent and
mathematically correct. They affect interoperability when comparing or
porting between the two.

A reproducible reference fixture (`Tests/MLXDiffRastTests/Fixtures/diffrast_fixtures.safetensors`)
was generated by running nvdiffrast on CUDA against a deterministic input.
The parity tests (`Tests/MLXDiffRastTests/ParityTests.swift`) load it and
compare to this library's output. The texture op matches **bit-exactly**;
the other gaps are explained by the conventions documented here.

---

## 1. Image-y orientation

| | This library | nvdiffrast |
|---|---|---|
| Pixel `(h=0, w=0)` maps to NDC | `(−1, +1)` (top-left) | `(−1, −1)` (bottom-left) |
| Image-y direction | down (PNG / image convention) | up (OpenGL convention) |
| Affected channels | rast (u, v, z, tri_id+1), rast_db (du/dy, dv/dy) | same |

**Symptom**: an identical mesh + camera produces images that are **vertical
flips** of each other.

**To compare**: apply `np.flip(image, axis=0)` (or `MLX.flip` on the H axis)
to one side. Note that `rast_db`'s `du/dy` and `dv/dy` channels also negate
under the flip because per-pixel-y direction reverses.

---

## 2. Barycentric channel layout in `rast`

The `rast` tensor's `(u, v)` channels store per-pixel barycentric weights,
but the two libraries pack which weights into which channels differently.

For a triangle with vertices `v₀, v₁, v₂` and barycentric weights
`w₀, w₁, w₂` (with `w₀ + w₁ + w₂ = 1`):

| | This library | nvdiffrast |
|---|---|---|
| `rast.u` | `w₁` | `w₀` |
| `rast.v` | `w₂` | `w₁` |
| Implicit (`bw = 1 − u − v`) | `w₀` | `w₂` |

**Symptom**: feeding our `rast` into nvdiffrast's `interpolate` (or vice
versa) produces interpolated values that are cyclic permutations across
the 3 attribute axes.

**Remapping** (use to convert between the two):

```python
# Convert nv rast → this library's rast:
my_rast = np.stack([
    1 - nv_rast[..., 0] - nv_rast[..., 1],   # u becomes w_1: was nv.v
    # actually:
    nv_rast[..., 1],                          # this lib's u = nv's v
    1 - nv_rast[..., 0] - nv_rast[..., 1],   # this lib's v = nv's bw
    nv_rast[..., 2],                          # z/w unchanged
    nv_rast[..., 3],                          # tri_id+1 unchanged
], axis=-1)
```

Or equivalently, the inverse:

```python
# Convert this library's rast → nv rast:
nv_rast_u = 1 - my_rast[..., 0] - my_rast[..., 1]   # nv.u = my.bw
nv_rast_v = my_rast[..., 0]                          # nv.v = my.u
```

## When neither convention difference matters

- **`texture`**: depends only on `(u, v)` passed in by the caller. Bit-exact
  match against nvdiffrast (verified by the parity tests).
- **Pure forward/backward optimization within this library**: rast →
  interpolate → texture → antialias → loss, all in our conventions, produces
  correct gradients. The inverse-rendering samples are evidence.
- **Visual output up to a vertical flip + channel permutation**: same.

## When it does matter

- Direct numerical comparison of `rast` / `interp` outputs against an
  nvdiffrast reference. (You'll need to apply both remappings.)
- Loading weights or fixtures generated under nvdiffrast's conventions.
- Porting Python nvdiffrast code that does explicit channel arithmetic on
  `rast` (e.g., extracting `bw = 1 - u - v` for some derived computation).

## Could these be changed?

Yes — both are 1–2 lines in the kernels. We chose not to:

- **Y-axis**: the image-top-left convention matches `CGImage`,
  `MTLTexture` reads, and how MLX users typically index `[N, H, W, C]`
  tensors (`h=0` is the top row). Matching this avoids a flip every time
  you save / display an output.
- **Barycentric layout**: this library's `(u, v) = (w₁, w₂)` choice means
  the implicit weight is `w₀ = 1 − u − v`, which lines up with the common
  "barycentric coordinates of the *other two* vertices" intuition used in
  graphics textbooks. nvdiffrast's choice has the opposite intuition.

If you want strict nvdiffrast compatibility, both conventions can be
toggled — open an issue / send a PR.
